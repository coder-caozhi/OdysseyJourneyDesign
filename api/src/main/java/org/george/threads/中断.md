# 一、先分两类核心场景，用代码直观演示
1. **synchronized 同步锁**：抢锁阻塞时 **不响应中断**
2. **ReentrantLock 可中断锁 `lockInterruptibly()`**：抢锁阻塞时 **能响应中断**
3. 顺带对比：`wait()` 等待也**能被中断**

---

# 场景1：synchronized 抢锁阻塞 —— 不响应中断
结论：线程卡在 **BLOCKED** 状态，调用 `interrupt()` **没任何反应**，不会唤醒、不会抛异常，只能死等拿到锁。

```java
public class SyncLockInterruptDemo {

    public static void main(String[] args) throws InterruptedException {
        // 同一把锁对象
        Object lock = new Object();

        // 线程1：先抢占锁，并且一直持有不释放
        Thread t1 = new Thread(() -> {
            synchronized (lock) {
                System.out.println("线程1 拿到锁，一直持有不释放");
                while (true) {
                    // 一直占用锁，不退出代码块
                }
            }
        });

        // 线程2：抢同一个synchronized锁，会进入 BLOCKED
        Thread t2 = new Thread(() -> {
            System.out.println("线程2 开始抢synchronized锁...");
            synchronized (lock) {
                System.out.println("线程2 终于拿到锁了");
            }
            // 这句永远没机会执行
            System.out.println("线程2 执行结束");
        });

        t1.start();
        Thread.sleep(500);
        t2.start();

        // 主线程休眠2秒，让t2进入 抢锁阻塞状态
        Thread.sleep(2000);
        System.out.println("主线程：调用 t2.interrupt() 尝试中断");
        // 尝试中断正在BLOCKED抢锁的线程
        t2.interrupt();

        // 打印线程状态：依然是 BLOCKED，毫无变化
        System.out.println("t2 线程当前状态：" + t2.getState());
        System.out.println("结论：synchronized 抢锁阻塞，不响应中断！");
    }
}
```
### 运行现象
- `t2` 一直卡在**BLOCKED**
- 调用 `interrupt()` 完全没效果，不会唤醒、不会报错
- 只要 `t1` 不释放锁，`t2` 永远卡死

# 场景2：ReentrantLock —— lockInterruptibly() 可中断抢锁
核心方法：`lockInterruptibly()` **抢锁过程中可以被中断，直接抛异常退出**

```java
import java.util.concurrent.locks.ReentrantLock;

public class ReentrantLockInterruptDemo {
    private static final ReentrantLock lock = new ReentrantLock();

    public static void main(String[] args) throws InterruptedException {

        // 线程1：先占用锁不释放
        Thread t1 = new Thread(() -> {
            lock.lock();
            try {
                System.out.println("线程1 拿到ReentrantLock，一直持有不释放");
                while (true) {}
            } finally {
                lock.unlock();
            }
        });

        // 线程2：用 lockInterruptibly() 尝试抢锁
        Thread t2 = new Thread(() -> {
            try {
                System.out.println("线程2 开始用 lockInterruptibly() 抢锁...");
                // 关键点：可中断的抢锁
                lock.lockInterruptibly();
                lock.unlock();
            } catch (InterruptedException e) {
                // 被中断后直接进到这里
                System.out.println("线程2 抢锁过程被中断了！抛出 InterruptedException");
            }
        });

        t1.start();
        Thread.sleep(500);
        t2.start();

        // 等2秒，让t2进入 等待锁 的阻塞状态
        Thread.sleep(2000);
        System.out.println("主线程：调用 t2.interrupt()");
        // 中断正在等锁的线程
        t2.interrupt();

        Thread.sleep(500);
        System.out.println("t2 线程当前状态：" + t2.getState());
        System.out.println("结论：ReentrantLock#lockInterruptibly() 抢锁可以被中断！");
    }
}
```
### 运行现象
- `t2` 在等锁过程中
- 主线程一调用 `interrupt()`
- **立刻抛出 `InterruptedException`**，线程终止，不再死等锁

# 场景3：Object.wait() 等待 —— 也能被中断
对应你之前生产者消费者代码里的 `wait()`，也能被 `interrupt()` 唤醒抛异常

```java
public class WaitInterruptDemo {
    public static void main(String[] args) throws InterruptedException {
        Object lock = new Object();

        Thread t = new Thread(() -> {
            synchronized (lock) {
                try {
                    System.out.println("线程进入 wait() 无限等待...");
                    // 无限等待 WAITING 状态
                    lock.wait();
                } catch (InterruptedException e) {
                    System.out.println("线程 wait() 被中断唤醒，抛异常");
                }
            }
        });

        t.start();
        Thread.sleep(2000);

        System.out.println("主线程调用 interrupt()");
        t.interrupt();
    }
}
```

# 一张表彻底记死
| 阻塞场景 | 是否可被 interrupt() 中断 | 表现 |
|--------|--------------------------|------|
| synchronized 抢锁（BLOCKED） | ❌ 不可中断 | 一直阻塞，无视中断 |
| ReentrantLock.lock() | ❌ 不可中断 | 同 synchronized |
| ReentrantLock.lockInterruptibly() | ✅ 可中断 | 立刻抛异常退出等待 |
| Thread.sleep() | ✅ 可中断 | 抛异常 |
| Object.wait() | ✅ 可中断 | 抛异常 |

# 关键一句话总结
1. **synchronized 抢锁**：死等，`interrupt()` 叫不醒；
2. **Lock#lockInterruptibly()**：可以被 `interrupt()` 叫醒，直接异常退出；
3. 你之前代码里的 `wait()`、`sleep()` 本身就自带可中断特性。

# interrupt() 的真正作用
先把两个核心问题给你**大白话讲透**，不用晦涩术语，再配结论。

---

## 一、先回答你最关键的疑问
### 1. `t2.interrupt()` 是不是直接把 t2 改成终止（TERMINATED）状态？
**❌ 完全不是！**
`interrupt()` **不杀死线程、不停止线程、不直接改状态为终止**。

### 2. 那「中断」到底是什么意思？
**中断 = 给线程打一个「中断标记位」 = 给线程发一个通知：有人让你该停下来了**
仅此而已，**不会强行杀线程**。

打个比方：
- `interrupt()` 只是**拍一下肩膀提醒你**
- 要不要停、什么时候停、怎么停，**由线程自己代码决定**

---

## 二、`interrupt()` 实际只做两件事
1. 把目标线程的**中断标志位**设为 `true`
2. 如果线程正处在**可中断阻塞**：
    - `sleep()` / `wait()` / `join()`
    - `lockInterruptibly()`
      会**立刻唤醒线程**，抛出 `InterruptedException`，同时**清空中断标记**

> 重点：
> 只是**唤醒 + 打标记 + 抛异常**，**不帮你结束线程**。

---

## 三、分三种情况，看 t2 调用 `interrupt()` 后会怎样

### 情况1：t2 正在正常运行（RUNNABLE）
- 调用 `t2.interrupt()`
- 仅仅**把中断标记置为 true**
- 线程**继续正常跑**，不会停、不会终止
- 除非你代码里主动判断：
  ```java
  if (Thread.currentThread().isInterrupted()) {
      return; // 自己退出，才会进入 TERMINATED
  }
  ```
**不判断、不退出，线程就一直跑。**

---

### 情况2：t2 在 `synchronized` 外抢锁（BLOCKED 状态）
就是你之前疑惑的：
- t2 卡在等 synchronized 锁，状态 BLOCKED
- 调用 `t2.interrupt()`
- **标记位被设为 true，但线程完全没反应**
- 依旧卡在 BLOCKED，不会醒、不会抛异常、不会终止
- 等后面抢到锁**开始执行代码了**，才能读到这个中断标记

---

### 情况3：t2 在 `sleep()` / `wait()` / lockInterruptibly()（WAITING/TIMED_WAITING）
- 调用 `t2.interrupt()`
- 立刻被唤醒、抛 `InterruptedException`
- 然后**看你 catch 里怎么写**：
    - 直接 return → 线程结束进入 TERMINATED
    - 继续循环跑 → 线程继续活着

**是你代码决定要不要终止，不是 interrupt() 强行终止。**

---

## 四、一句话直白定义
1. **中断（interrupt）**：
   只是**设置一个中断标记、唤醒可阻塞线程**，**不杀线程**。
2. **`t2.interrupt()` ≠ 让 t2 立刻死亡**
3. 线程进入 **TERMINATED 终止状态**，只有一种可能：
   **线程自己的代码执行完毕 / 主动退出**，和 interrupt() 没有直接关系。

---

## 五、极简总结背下来
- `interrupt()`：**发通知、打标记、唤醒阻塞线程，不杀线程**
- 不会直接把线程变成 TERMINATED
- 要不要退出、什么时候退出，**线程自己代码说了算**
- synchronized 抢锁 BLOCKED：无视中断，继续死等
- sleep/wait/lockInterruptibly：被中断会立刻唤醒抛异常
